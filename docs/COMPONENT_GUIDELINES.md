# Component Guidelines

AIR components should be reusable, data-driven, and independent from Playwright-specific assumptions.

## Component Responsibilities

| Component Type | Responsibility |
| --- | --- |
| Badge | Display compact semantic status without overflow |
| KPI card | Display one metric and short context |
| Module card | Summarize module health and open module details |
| Journey node | Summarize journey step health and open related module |
| Evidence tile | Preview available artifact |
| Empty state | Explain unavailable data with title, reason, and next action |
| Tooltip | Explain metrics using metadata-backed copy |
| Section header | Introduce page sections with consistent title and context |
| Status indicator | Show healthy, warning, critical, planned, or no-data state |
| Progress bar | Display bounded percentage progress |
| Table | Provide advanced detail after summary cards |
| Drawer | Show focused drill-down without leaving page |
| Modal | Explain a metric or preview evidence |

## Component Rules

- Accept normalized AIR data.
- Do not calculate business rules inside visual components.
- Do not duplicate information unless the page purpose requires it.
- Keep states explicit: healthy, warning, critical, planned, no data.
- Make interactive cards keyboard accessible.
- Do not hardcode Playwright-specific labels inside reusable UI pieces.
- Prefer semantic status classes or data attributes over one-off styling.
- Support long labels, especially `CONDITIONAL GO`, without fixed-width assumptions.

## Release Badge Component

Use the shared release badge helper for every release decision.

Required behavior:

- Normalize `CONDITIONAL_GO` and `CONDITIONAL GO` to the same display.
- Normalize `NO_GO` and `NO GO` to the same display.
- Emit one semantic `data-status` value.
- Remain readable inside cards, headers, modals, and sidebars.
- Use responsive font sizing and wrapping instead of hardcoded widths.

## Tooltip Component

Tooltips should read from the tooltip metadata layer.

Current default metadata keys:

- `qualityScore`
- `releaseDecision`
- `risk`
- `coverage`
- `recommendation`
- `businessHealth`
- `evidenceReadiness`
- `nextStep`

If AIR Core later provides dynamic explanation text, the UI may override the default metadata without changing component structure.

## Empty State Component

Use one empty-state shape:

| Field | Required |
| --- | --- |
| Title | Yes |
| Reason | Yes |
| Next Action | Yes |
| Icon | Optional |
| Metrics | Optional |

Do not show failure-colored cards for missing or unavailable data unless the missing data itself is a failure.

## Card and Grid Rules

- Peer cards should use the same gap and padding within a section.
- Cards must tolerate long labels, wrapped badges, and missing values.
- Use responsive grids: desktop can show multiple cards, tablet fewer, mobile one column.
- Avoid nested cards unless the child is an explicit metric, evidence tile, modal, or drawer section.
- Keep summary cards short; move detail into drawers, modals, tables, or drill-down sections.

## Lightweight UI Foundation

Current reusable foundations are implemented as helpers and shared CSS inside `scripts/generate-execution-report.js`:

- Release badge formatter and renderer.
- Tooltip metadata and `helpLabel`.
- Empty state renderer.
- Design token aliases for status colors, spacing, and typography.

When AIR moves to a component-based UI, these foundations should become first-class components without changing the visual language.

## Current Implementation Note

The current dashboard is generated by `scripts/generate-execution-report.js`. As AIR evolves, reusable rendering helpers should be extracted before adding complexity.
